# Grafana Dashboard — Implementation Notes

## How to Import

1. In Grafana, go to **Dashboards > Import**
2. Upload `grafana-dashboard.json` or paste its contents
3. Select your **Prometheus data source** when prompted
4. Click **Import**

Access TV mode: append `?kiosk` to the dashboard URL.

## What Needs to Change for Real Data

### 1. Namespace Values (MUST CHANGE)

Every query uses hardcoded namespace filters. Replace these with your actual Kubernetes namespace names:

| Placeholder | Replace with |
|------------|--------------|
| `system-a-prod` | Your actual namespace for System A |
| `system-b-prod` | Your actual namespace for System B |
| `system-c-prod` | Your actual namespace for System C |
| `system-d-prod` | Your actual namespace for System D |

Search and replace across the JSON file.

### 2. Job Names for Uptime Query (MUST CHANGE)

The Uptime/SLA gauge uses `up{job="system-a", ...}`. The `job` label value depends on your Prometheus scrape config — typically the service name or the ServiceMonitor name.

Run this to find your actual job names:
```promql
group by (job) (up{namespace="your-namespace"})
```

Replace `job="system-a"`, `job="system-b"`, etc. with the correct values.

### 3. Row Titles (SHOULD CHANGE)

Row headers use placeholder names like `System A — eks-prod-apse1 / system-a-prod`. Replace with:
- Your actual system/application names
- Your actual EKS cluster name
- Your actual namespace

### 4. SLA Targets (SHOULD CHANGE)

SLA threshold values are set per system in the Uptime/SLA gauge panels:

| System | Current Threshold | Panel ID |
|--------|------------------|----------|
| System A | 99.95% | 102 |
| System B | 99.90% | 202 |
| System C | 99.95% | 302 |
| System D | 99.90% | 402 |

Change the threshold `value` in each gauge panel's `fieldConfig.defaults.thresholds.steps` to match your agreed SLA targets.

### 5. Latency Thresholds (MAY CHANGE)

Current latency thresholds (in seconds):
- Green: < 100ms (0.1s)
- Yellow: 100ms–500ms
- Red: > 500ms

Adjust per system if they have different latency profiles (e.g., a batch system may tolerate higher latency).

### 6. HTTP Metric Names (MAY CHANGE)

Queries assume standard Prometheus metric names:

| Metric Used | Comes From | Alternative Names |
|------------|-----------|-------------------|
| `http_request_duration_seconds_bucket` | App instrumentation | `http_server_request_duration_seconds_bucket` (OpenTelemetry), `request_duration_seconds_bucket` |
| `http_requests_total` | App instrumentation | `http_server_requests_total` (OTel), `requests_total` |
| `container_cpu_usage_seconds_total` | cAdvisor/kubelet | Usually consistent |
| `container_memory_working_set_bytes` | cAdvisor/kubelet | Usually consistent |
| `kube_deployment_status_replicas_ready` | kube-state-metrics | Usually consistent |
| `kube_deployment_status_replicas` | kube-state-metrics | Usually consistent |
| `kube_pod_container_resource_limits` | kube-state-metrics | Usually consistent |

If your apps use OpenTelemetry instrumentation, the metric names may differ. Check with:
```promql
{__name__=~"http.*", namespace="your-namespace"}
```

### 7. Alerts Metric (MAY CHANGE)

The Critical Alerts panel queries `ALERTS{alertstate="firing", severity="critical"}`. This requires:
- Prometheus alerting rules configured with `severity: critical` label
- The `ALERTS` metric is auto-generated by Prometheus when rules are active

If you use a standalone Alertmanager without Prometheus ruler, switch to the **Alertmanager data source** and use its native alert query instead.

## Prerequisites (Must Be Running)

| Component | Required For | How to Verify |
|-----------|-------------|---------------|
| **kube-state-metrics** | Health, CPU/Memory queries | `kubectl get pods -n kube-system -l app.kubernetes.io/name=kube-state-metrics` |
| **cAdvisor** (built into kubelet) | Container CPU/Memory | Always present on EKS |
| **Prometheus** (AMP or self-hosted) | All queries | Check data source in Grafana |
| **Application instrumentation** | Latency, Error Rate, Throughput | App must expose `http_request_duration_seconds_bucket` and `http_requests_total` metrics |
| **Alerting rules** | Critical Alerts panel | `ALERTS` metric must exist in Prometheus |

### Common Missing Pieces

**No `http_request_duration_seconds_bucket`?**
Your app needs a histogram metric for request latency. If using:
- Spring Boot: add `micrometer-registry-prometheus` (metric: `http_server_requests_seconds_bucket`)
- Node.js: use `prom-client` with histogram
- Go: use `promhttp` handler
- OpenTelemetry: auto-instrumentation provides this

**No `http_requests_total`?**
Same as above — the counter for total HTTP requests must be exposed.

**CPU/Memory shows 0% or N/A?**
Likely missing resource limits on pods. The CPU/Memory query divides usage by `kube_pod_container_resource_limits`. If pods have no limits set, the denominator is 0. Fix by setting resource limits in deployment manifests, or change query to use `kube_pod_container_resource_requests` instead.

## Uptime/SLA Query — Important Notes

The current query `avg_over_time(up{...}[30d])` has caveats:

1. **Prometheus retention**: Must retain 30 days of data. Default is 15 days. Check with `--storage.tsdb.retention.time` flag.
2. **Performance**: `avg_over_time(...[30d])` scans 30 days of data on every evaluation. For production, create a **recording rule** instead:
   ```yaml
   groups:
     - name: sla
       interval: 1m
       rules:
         - record: system:availability:ratio_30d
           expr: avg_over_time(up{job="system-a"}[30d])
   ```
3. **Better SLA metric**: Consider using blackbox_exporter for true external availability:
   ```promql
   avg_over_time(probe_success{job="blackbox", target="https://system-a.internal"}[30d]) * 100
   ```
4. **What `up` actually measures**: The `up` metric tells you if Prometheus can scrape the target. It doesn't measure user-facing availability. For true SLA, use synthetic monitoring or error-rate-based SLA.

## TV Display Tips

- **Kiosk mode**: Append `?kiosk` to URL (hides sidebar + navbar) or `?kiosk=tv` for full kiosk
- **Auto-refresh**: Set to 30s (already configured in the dashboard JSON)
- **Browser**: Use Chrome in full-screen mode (F11)
- **Font size**: If text is too small at 3-5m viewing distance, reduce to 5 columns (drop CPU/Memory and move alerts to its slot, or merge Health + Uptime)
- **Playlist mode**: If you have multiple dashboards, use Grafana Playlist to rotate between them
